# AIC Compliance & Regulations Guide

> **Purpose:** This document is a strict, actionable guideline extracted from the official AI for Industry Challenge (AIC) repository. Every rule below must be followed to avoid disqualification, score penalties, or prize revocation. When in doubt, the answer is **don't do it**.

---

## 1. Competition Spirit & Core Principle

The AIC is designed to foster creativity and technical excellence in robotic cable insertion. All solutions **must** use only the officially provided ROS 2 interfaces (topics, services, and message types defined in `aic_interfaces`). The goal is to build AI models that are **generalizable, robust, and compliant** with the constraints of the physical system and its simulation.

**Golden Rule:** If the interface isn't listed in the official AIC Interfaces documentation, you cannot use it.

---

## 2. Prohibited Actions — What Constitutes Cheating

### 2.1 Direct State Manipulation
**FORBIDDEN:**
- Manually altering the robot, task board, or simulation environment state
- Teleporting components or forcing cable insertions programmatically
- Writing to any simulation entity's pose, velocity, or physics state

### 2.2 Backend & System Interference
**FORBIDDEN — Do NOT interact with any of the following:**
- **Configuration & Lifecycle:** Modifying ROS 2 parameters or changing node lifecycle states for evaluation nodes (you may only manage your own `aic_model` node's lifecycle)
- **Simulation Control:** All topics/services in the `/scoring`, `/gazebo`, and `/gz_server` namespaces
- **Entity Management:** Services used to spawn, despawn, or delete models and entities
- **Environment States:** Data related to `/clock`, `/model`, `/world_stats`, `/pause_physics`, or simulation resets
- **Any Gazebo backend service** not explicitly listed as a participant interface

### 2.3 Exploitative Hardcoding
**FORBIDDEN:**
- Hardcoding sensor data or environment configurations to exploit knowledge of the specific simulation setup
- Embedding fixed pose targets derived from prior knowledge of exact board/port locations (the board pose, component positions, and orientations are randomized per trial)
- Using static waypoints that only work for one specific trial configuration

### 2.4 Process Manipulation
**FORBIDDEN:**
- Interfering with the AIC Engine (`aic_engine`) in any way
- Tampering with, disabling, or modifying the scoring system (`aic_scoring`)
- Interfering with the logging infrastructure

### 2.5 Unauthorized Access to Evaluation Infrastructure
**FORBIDDEN:**
- Reverse-engineering, exploiting, or tampering with the cloud-based evaluation infrastructure
- Using unauthorized tools to access or modify the evaluation environment
- Submitting containers containing malicious code, backdoors, or mechanisms to defeat the competition architecture
- Bypassing the submission portal or registry to inject or manipulate data
- Attempting to crack or bypass the Zenoh ACL password-protected `eval` identity

### 2.6 External Data & Information Leaks
**FORBIDDEN:**
- **State Leaking:** Reusing internal simulation/backend state information **during evaluation** that is not available through official interfaces
- **Data Misuse:** Training models using data generated by the evaluation infrastructure that is not part of the toolkit

**ALLOWED during training only:**
- Using all internal state information including ground truth data available over the `/tf` topic (set `ground_truth:=true`)
- The `CheatCode` example policy demonstrates this — it is explicitly for training/debugging only and will NOT work during evaluation

- `/gz_server/*`
- `/clock`
- `/model/*`
- `/world_stats`
- `/pause_physics`

## 4. Node Behavioral Requirements — `aic_model` Lifecycle

Your submission container must start a ROS 2 Lifecycle node named `aic_model` that follows this exact behavioral contract:

### 4.1 State Machine

| State | Requirements | Timeout |
|-------|-------------|---------|
| **Unconfigured** | No topics published. No robot commands. | — |
| **Configured** (via `configure`) | No topics published. No robot commands. Reject any `/insert_cable` goal requests. | 60s to complete transition |
| **Active** (via `activate`) | Accept `/insert_cable` goals. Goals must be cancellable. Complete goals within the `time_limit` field from the Task message. | 60s to complete transition |
| **Configured** (via `deactivate`) | Return to configured behavior. | 60s to complete transition |
| **Unconfigured** (via `cleanup`) | Return to unconfigured behavior. | 60s to complete transition |
| **Shutdown** (via `shutdown`) | No topics published. No robot command publishers present in the ROS graph. | 60s to complete transition |

### 4.2 Critical Lifecycle Rules
- **Loading models** (e.g., neural network weights) must happen within the `configure` transition's 60-second window
- In **configured** state, the node must actively **reject** `/insert_cable` goals
- In **active** state, the node must **accept** goals and they must be **cancellable**
- In **shutdown** state, there must be **zero** robot command publishers on the graph
- All state transitions must succeed within **60 seconds**

## 5. Scoring Penalties — How to Lose Points

Understanding penalties is critical to avoiding rule-related score losses.

### 5.1 Model Validity (Tier 1) — Pass/Fail Gate
- If your model fails to activate, respond to InsertCable, or send valid commands → **0 points, no further scoring**
- Must comply with ALL behavioral requirements in Section 4

### 5.2 Insertion Force Penalty (Tier 2): 0 to −12 points
- Force threshold: **20 N**
- Duration threshold: **1 second**
- If force exceeds 20 N for more than 1 second → **−12 points**
- The F/T sensor is tared at startup (baseline ≈ 0 N)

### 5.3 Off-Limit Contact Penalty (Tier 2): 0 to −24 points
- Any contact with restricted areas (enclosure or task board body) → **−24 points**
- This is binary — one collision costs you the full penalty

### 5.4 Wrong Port Insertion (Tier 3): −12 points
- Inserting into the **wrong** port → **−12 points** (instead of +75 for correct)

### 5.5 Maximum Score Per Trial: 100 points
- Tier 1 (validity): 0–1
- Tier 2 (performance): up to +24, down to −36
- Tier 3 (insertion): up to +75, down to −12

## 6. Submission Rules

### 6.1 Container Requirements
- OCI-compliant container (Docker/Podman)
- Must use `rmw_zenoh_cpp` as ROS 2 middleware (set via `RMW_IMPLEMENTATION` env var — do not override it)
- Must connect to the Zenoh router provided via `AIC_MODEL_ROUTER_ADDR`
- Must authenticate with user `model` and password from `AIC_MODEL_PASSWD`
- ROS 2 distribution: **Kilted Kaiju** (official evaluation distro — inter-distro communication is NOT guaranteed)

### 6.2 Submission Limits
- **1 submission per day** per team (applies to the whole team, not per individual)
- Image tags in ECR are **immutable** — you must increment version tags (`:v1`, `:v2`, etc.)
- Any team member can submit on behalf of the team
- The **final submission** before deadline is used for scoring

### 6.3 Pre-Submission Checklist
1. Verify locally with `docker compose -f docker/docker-compose.yaml up`
2. Test with Zenoh access controls enabled (see `access_control.md`)
3. Confirm your node passes all lifecycle transitions within timeouts
4. Confirm your container does NOT access any blocked topics/services
5. Confirm no hardcoded poses or simulation-specific exploits
6. If a container fails to start or crashes during local evaluation, it will be **automatically rejected** and counts against daily submission limit

## 7. Enforcement — How They Catch Violations

### 7.1 Automated Checks
- **Zenoh Access Control Lists (ACL):** The evaluation environment uses Zenoh ACLs to block access to simulation internals. Topics like `/gz_server/*` are blocked for the `model` identity. The `eval` identity is password-protected with a different password in production.
- **Automated integrity checks** run as part of the evaluation pipeline

### 7.2 Manual Review (Top-Performing Teams)
- **Container Audits:** Submitted container images are reviewed to verify they only interact via defined interfaces
- **Behavioral Verification:** Model behavior during live evaluations is manually inspected
- **Metric Analysis:** Performance is cross-checked against expected baselines to detect anomalies (e.g., suspiciously perfect scores or impossible trajectories)

### 7.3 Consequences
- **Disqualification** from the competition
- **Revocation of prizes**
- Reports from other participants are treated confidentially

## 8. What IS Allowed — Safe Practices

### 8.1 Training
- Use **any simulator** for training: Gazebo, MuJoCo, Isaac Sim, O3DE, etc.
- Use **real-world teleoperation data**
- Use **classical control algorithms**
- Use ground truth data (`ground_truth:=true`) during training — this is encouraged for debugging and data collection
- Use domain randomization across simulators
- Use the provided LeRobot integration for data collection and ACT training
- Use the `CheatCode` example policy for training/debugging (it uses ground truth)

### 8.2 Policy Development
- Any ML framework (PyTorch, ONNX, TensorFlow, etc.)
- Any training approach (imitation learning, reinforcement learning, classical control, hybrid)
- Custom ROS 2 nodes (as long as they meet all `aic_model` behavioral requirements)
- Additional Python/conda/pip dependencies in your container

### 8.3 During Evaluation
- Subscribe to all topics listed in Section 3.1
- Publish to topics listed in Section 3.2
- Call services listed in Section 3.3
- Use `/tf` and `/tf_static` for coordinate transforms (note: ground truth frames are NOT published during evaluation)
- Process camera images, joint states, force/torque data, and controller state

## 10. Quick Decision Checklist

Before implementing any approach, ask yourself:

| Question | If YES |
|----------|--------|
| Does it use a topic/service not in the whitelist (Section 3)? | **STOP — violation** |
| Does it read from `/scoring`, `/gazebo`, `/gz_server`, or `/clock`? | **STOP — violation** |
| Does it hardcode poses or exploit a specific simulation layout? | **STOP — violation** |
| Does it modify evaluation node parameters or lifecycle states? | **STOP — violation** |
| Does it spawn, despawn, or delete any entities? | **STOP — violation** |
| Does it use ground truth data at evaluation time? | **STOP — violation** |
| Does it tamper with the scoring or logging system? | **STOP — violation** |
| Does the node publish commands in unconfigured/configured/shutdown states? | **STOP — violation** |
| Does it fail to make goals cancellable in active state? | **STOP — violation** |
| Does it take longer than 60s for any state transition? | **STOP — violation** |

If all answers are NO, you're likely compliant. When in doubt, use only what's explicitly documented as allowed.
